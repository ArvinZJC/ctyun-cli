# AGENTS.md

## Project Shape
- This repo is `ctyun-cli`; the shipped command name is `ctyun`. Use `ctyun-cli` for repository/package context and `ctyun` for command examples, binaries, and help text.
- `cmd/ctyun/main.go` is intentionally tiny. Most behaviour enters through `internal/cli.Run`/`Execute`, which parses global flags, loads config/profile state, resolves language, loads plugin bundles, and dispatches either core commands or metadata-defined product commands.
- Product commands must come from plugin bundle metadata, not hardcoded product branches in `internal/cli/cli.go`. Existing bundles under `plugins/` define `plugin.json`, `commands.json`, `apis.json`, `tables.json`, `waiters.json`, fixtures, and `i18n/*.json`.
- `internal/openapipipeline` owns the dev-only OpenAPI catalog pipeline, and `tools/openapi` is only a repository-local maintenance entrypoint. The pipeline works from normalized catalogs under `openapi-catalogs/<name>/`, keeps generated drafts separate from promoted plugin metadata, and advances `baseline.json` when a review-ready draft is promoted, including generated-quality drafts when that is the truthful quality level. Track durable evidence (`source.json` and promoted `baseline.json`); leave reproducible local review outputs (`draft/`, `changes.md`, and `review.md`) ignored and regenerate them when reviewing. Do not expose this pipeline as a public `ctyun` command or package OpenAPI catalog artifacts into core/plugin release outputs.
- `tools/plugincheck` owns repo-local plugin release verification. Keep real plugin fixture/offline smoke coverage there instead of adding product-specific command tests under `internal/cli`; core CLI tests should use synthetic plugin metadata for command-engine behaviour.
- Live API execution flows through `internal/client` and `internal/signing`: command metadata resolves profile values, path args, and flags into request query/body/header fields, then signs with CTyun EOP headers.
- `internal/output` owns table rendering and raw JSON output. Table definitions still use stable column keys such as `instance_id` or `region_id`, while user-facing table controls (`--cols`, `--filter`, `--sort`) should accept visible localized column labels as well as stable keys. Column selection, filtering, and sorting should operate on what table users see; when row values are localized for display, filter and sort against those localized values. Never treat raw CTyun OpenAPI field names as table-control names unless they are also declared table keys.
- Before adding or changing user-facing CLI text, check the i18n support bullet in `README.md`/`README-EN.md` for the public localization scope. Use `helpText`/`helpf` only for help pages, help labels, option descriptions, and help-only hints. Use `messageText`/`messagef` for runtime warnings, notices, status lines, diagnostics, and error templates. Use `commonText` only for shared formatting tokens that are neither help nor runtime copy. Lower-level packages that cannot know the active language should return diagnostic message keys and arguments instead of hard-coded English templates. Implement covered plugin text through plugin i18n/table metadata, and avoid hard-coded English-only output in runtime paths. Do not add new `if language == "zh-CN" { ... } else { ... }` branches for user-facing text; add catalog entries instead.
- Keep `internal/cli/messages.go` style category-based: `error.*` entries should be diagnostic fragments without terminal punctuation, standalone warnings/notices/status messages should be sentence style with terminal punctuation, usage strings should keep `Usage:` syntax without a final period, and enum/status tokens should stay compact labels without punctuation.
- Do not expose internal command IDs such as `job.info` or `ecs.instance.list` in common-user-facing help, warnings, errors, examples, or runtime diagnostics. Use the visible command path, visible option/argument names, plugin/product names, or a generic "plugin command" subject instead. Metadata files, plugin lint diagnostics, tests, and dev-only OpenAPI review output may mention command or operation IDs when they are specifically reporting catalog metadata.
- Treat generated shell completion scripts, command syntax/examples, raw JSON, CTyun OpenAPI request/response fields, checksums, keys, paths, and other machine-oriented data as data rather than prose; do not translate those values. Table-control selector values inside user-facing examples are the exception: render declared table keys as visible localized column labels when help output can resolve the command table. Translate the surrounding help, status, warning, diagnostic, and error prose that explains machine-oriented values.
- Core and plugin command help should follow one grouped-command shape: a concise sentence description, `Usage`, `Subcommands` when applicable, `Arguments` for positional placeholders, `Command Options` only for options owned by that command, and shared `Global Options`. Usage lines should include `[global options]`, show required command options without brackets, and show optional command options in brackets; conditionally required plugin options stay optional in usage and carry the condition in option help. Keep command option value placeholders as `<value>` or `<A|B|C>` without repeating the same value set in prose. Keep compact command, subcommand, argument, and option list descriptions punctuation-free; render page-leading standalone descriptions as sentences. Sort non-positional help rows for stable scanning, but preserve argument order. For plugin table selectors, render `Columns` or `Fields` one selector per line and put default markers in a separate right-side description column. Do not add flags that only change formatting unless they have a distinct semantic purpose.
- Keep command parsing, help usage, option help, applicability, and completion driven by shared command/option declarations rather than parallel command-specific lists. A valid command group with no child renders that group's help and exits zero. At a command boundary, classify an undeclared option-shaped token as generic `unknown option` and an unknown word as generic `unknown command` with the visible command path; after a complete leaf, classify an extra positional token as `unexpected argument`, and classify a missing positional token as `missing required argument`. Do not add redundant subjects such as `config profile subcommand`, `upgrade option`, `plugin list option`, or `doctor network option`. Use `{name}` for positional placeholders and `<name>` for option values.
- Development-only options are command-owned rather than global and must use long names only. `--offline` and `--fixture` are accepted only after a complete metadata-defined product command path; `--bundled` is accepted only by its supported plugin-management commands. Do not add or preserve short aliases such as `-O`. Released builds must treat these spellings like ordinary unknown options and must not expose development-option help, completion, or special diagnostics.
- When an option has a fixed default value, expose it through the shared option-help metadata/formatter instead of hand-writing default text in individual descriptions.
- `internal/cli/completion.go` owns shell script generation and the hidden `__complete` resolver. When changing core commands, plugin subcommands, global flags, metadata-defined command paths, parameter-allowed values, table column keys, waiters, or supported shells, keep completion behaviour and tests in sync.
- `internal/distribution` owns hosted source resolution, index/artifact fetching, checksum verification, signature verification, and shared artifact URL/path safety. `internal/registry` owns plugin registry metadata validation/search, and `internal/plugin/install.go` owns local plugin bundle install safety: safe plugin names, compatibility checks, `.tar.gz` extraction, and traversal/symlink rejection.

## Workflows
- Do not store local absolute directories in docs or code. Use repo-relative paths or `$PWD`; if an absolute path is truly required and the session does not expose a valid repo root, ask the user for the path first.
- Keep GitHub-facing text such as pull request descriptions, release notes, issue comments, and review replies free of local absolute paths, user directories, temporary file paths, secrets, tokens, and machine-specific details. Use repo-relative paths, public URLs, version or tag names, and command examples instead.
- Write prose as natural Markdown paragraphs and list items. Do not hard-wrap prose at a fixed column; preserve line breaks only where Markdown structure or readability requires them, such as headings, lists, tables, block quotes, and code blocks.
- Keep public usage, installation, auth/config/language, registry/plugin/update, fixture/offline, and contributor workflow wording in `README.md`/`README-EN.md`; use those files for dev, debug, test, coverage, and plugin-lint commands instead of duplicating command lists here.
- `README.md` is Simplified Chinese and `README-EN.md` is the English counterpart; when changing user-facing README content, keep both files semantically in sync.
- Versioning policy is public README content. Core and plugin versions must follow the SemVer policy documented there; keep release tooling, plugin metadata, and examples aligned with it.
- Maintain canonical changelogs in this repo. Use the root `CHANGELOG.md` for core `ctyun` releases, and plugin-local `plugins/<name>/CHANGELOG.md` files for independently tracked plugin release history; if a changelog grows too large, move older entries into versioned archive files such as `changelogs/1.x.md` and link them from the active changelog. Each `Unreleased` or version section must describe changes relative to the immediately preceding release of that core or plugin; do not restate cumulative capabilities or unchanged behaviour.
- When core or plugin release metadata changes, update the matching changelog and public README tables/examples in the same pass. For plugin manifest or command-surface changes, keep `plugins/<name>/plugin.json`, `plugins/<name>/CHANGELOG.md`, and the README/README-EN plugin table channel, quality, command count, and operation count aligned. For core version changes, keep `internal/version/version.go`, `CHANGELOG.md`, release-tool examples, and compatibility-sensitive tests aligned.
- When the user asks for detailed user, developer, or plugin-author documentation, create it under the project-root `docs/` directory unless they specify another tracked path. Do not use ignored planning folders for public docs unless explicitly asked.
- When touching Go code, add or improve Go doc comments for all package-level functions, types, interfaces, variables, constants, and package-level behaviour. Keep comments factual and useful; avoid filler comments on obvious implementation steps.
- When reducing duplication, extract shared behaviour into the package that owns the abstraction and have callers use it directly. Avoid one-line proxy functions whose body only forwards to another function; keep a wrapper only when it adds package-specific behaviour, preserves a deliberate public API, or documents an intentional test seam.
- Keep each Go code file, including tests, under 1000 lines. Split helpers, command groups, or package modules before a file crosses that boundary.
- After every source or test change, run code formatting, the source-file line-limit check, and the coverage check from the README workflow before handoff. Treat all three as required source-change verification steps.
- For docs-only or config-only changes, run lightweight checks such as `git diff --check` unless the change affects runtime behaviour or developer commands.
- Commit subjects must use one of these prefixes only: `feat:`, `fix:`, `docs:`, `style:`, `build:`, `refactor:`, `revert:`, `test:`, `perf:`, `ci:`, or `chore:`. Keep the subject in a concise non-sentence style without trailing punctuation.

## File Headers
- DTD, Go, HCL, HTML, shell script, SQL, SVG, XHTML, and XML source headers are generated by GoLand copyright profiles. Preserve existing generated year ranges instead of hand-normalising them.
- Preserve required leading interpreter or document declaration lines such as shell shebangs and XML declarations; the generated header should sit in the language-appropriate comment position after those required lines.
- Use this header template when a source file needs the generated copyright header:

```text
Copyright (c) $originalComment.match("Copyright \(c\) (\d+)", 1, "-", "$today.year")$today.year IsArvin.
This file is part of $project.name. Please refer to the LICENCE file for licence information.
```

## Config, Credentials, And Live Calls
- Public CLI behaviour for auth/config/language, config commands, registry use, plugin installation, and fixture/offline modes lives in `README.md`/`README-EN.md`; consult those sections before related changes instead of copying those rules here.
- Live verification should stay on safe retrieval paths such as `region list` or ECS list/show, using ephemeral credentials when possible. Use `--offline` or `--fixture` when testing bundled fixtures.
- `--debug` writes redacted HTTP diagnostics to stderr; preserve the existing redaction path for AK/SK, request IDs, and signatures.

## Plugin Conventions
- Add product coverage by adding/reviewing plugin metadata and tests, not by adding `case "ecs"`-style dispatch. Command paths may use simple words and `{argument}` placeholders.
- `plugin.json` requires `stable|beta|alpha` channels, `generated|reviewed|curated` quality, CTyun version constraints, product metadata, and an HTTPS `api.endpoint_url` when live execution is supported. Choose the highest truthful quality level; use the README plugin table for the public meaning of each quality value.
- In plugin and OpenAPI catalog metadata, `api.ctyun_product_id` / `product.ctyun_product_id` comes from the CTyun OpenAPI docs `sid` query parameter, while `api.source_revision` / `product.source_revision` comes from `vid` when upstream exposes it. Treat `source_revision` as weak provenance. Future generated plugins should start from tracked normalized source catalogs under `openapi-catalogs/<name>/source.json`; promote generated-quality drafts only when `generated` is the truthful release quality, and promote beyond `generated` only when the manifest `source_fingerprint` matches that tracked source evidence.
- OpenAPI-maintained plugins should declare their upstream API selection boundary with `product.api_scope` in `openapi-catalogs/<name>/source.json` and `api.api_scope` in `plugins/<name>/plugin.json`, for example URI prefixes such as `/v4/ecs/` or `/v4/region/`. Do not silently generate, review, promote, or release APIs outside the declared scope; if a plugin needs multiple upstream ranges, list each prefix explicitly.
- `commands.json` binds a command ID/path to an operation, table, optional flags, fixture, docs URL, examples, and dangerous confirmation. Product commands use one canonical path; aliases are not part of the metadata model. Dangerous commands prompt for confirmation by default; `--yes`/`-y` skips the prompt.
- If official online docs still publish an API, command option, or response field but mark it as deprecated, obsolete, or planned for shutdown, keep the corresponding command, parameter, or table column in plugin metadata. Record `deprecation` metadata on the operation, command, parameter, or table column so help can mark it and runtime use can warn. Preserve upstream notices and upstream API replacement hints in metadata when useful, but show replacement guidance to users only when metadata names a CLI-facing command or option replacement.
- `apis.json` maps `$profile.region`, `$arg.<name>`, and `$param.<name>` into request fields. Retrieval operations may set `retryable`; state-changing operations should not unless metadata explicitly opts in.
  Accepted non-800 CTyun status metadata must be guarded by a response path that proves the response still contains useful command result data, such as `returnObj.satisfied`; never use normal error-envelope fields such as `error` or `errorCode` as accepted-status guards.
- Treat API `regionID` as a source-resolved request field, not as a product resource identity, except where the Region plugin itself exposes a resource-pool command. Generated plugin commands that map `regionID` from `$profile.region` must expose an optional `--region` command option targeting `regionID`; the local requirement check should fail only when neither profile `region` nor the exposed command input provides it. Commands that expose `{region_id}` must not also expose a duplicate `--region` option; Region commands may keep trailing `{region_id}` for compatibility and allow profile fallback when that trailing argument is omitted.
- `tables.json` defines display row paths and stable column keys. If a command flag maps to a table column target, fixture output is locally filtered the same way the live request is parameterised.
- Add localized help text under plugin `i18n/*.json` and table labels under `tables.json`; do not localize CTyun OpenAPI field names or payload values in plugin metadata. User-facing plugin examples should come from captured official OpenAPI examples; when an example includes a path placeholder, fill it from captured upstream example data instead of inventing values, and leave it as a placeholder only when no official value is available yet.
